# Weekly Newsletter — MVP Scope

**Status:** Scoped 2026-04-21. Not yet built.
**Supersedes:** the build-order section of `memory/project_newsletter_build.md` for v1 only. Design decisions in that memory note (draft-first philosophy, Canva preference, game concept, brand voice) still stand for v2+.

---

## Goal

Ship a weekly email to `subscribers` that (a) survives weeks when Jill is traveling without manual action, (b) proves the generation → send loop end-to-end, (c) stays narrow enough to ship in 1–2 build sessions.

## Cadence

- **Generate:** Thursday 6:00pm ET (cron)
- **Send:** Friday 7:00am ET (cron)
- **Admin preview:** Thursday 6:05pm ET — single email to Jill with "Edit" and "Hold this week" links

## Default-send model

Flips the default from "Jill must click send" to "Jill must click hold." Reliable when Jill is offline.

Statuses on `newsletters` row:
- `scheduled` — draft written, awaiting Friday send
- `held` — Jill clicked "Hold this week"; Friday cron skips
- `sent` — Friday cron sent it; `sent_at` + `recipient_count` populated
- `failed` — Resend error; flagged in admin

Friday 7am cron logic: send only rows where `week_of = this_week` AND `status = 'scheduled'`.

## Sections (3)

1. **Hero** — top alert of the week (1)
2. **Deal highlights** — 2–3 items from published alerts, last 7 days
3. **Tip or Take** — 1 item pulled from top-priority `content_ideas`

**Length target:** 600–900 words. Scannable. No images v1.

## Subject line

Pattern: `"Crazy Friday, Jill. <hook>"` — weekly variant of the daily brief's `"Crazy time, Jill."` tagline ([utils/ai/generateEditorialPlan.ts:272](../utils/ai/generateEditorialPlan.ts:272)).

Hook is a rotating sassy one-liner generated by Sonnet from the week's content. No A/B in v1.

## Generator input

Use upstream signals the daily brief is already producing. Don't re-invent selection logic.

- **Deal candidates:** `content_ideas` rows where `type = 'newsletter'` from the last 7 days (these are the daily brief's `newsletter_candidates`, already filtered to approved + deadline-aware by the editorial plan — see [utils/ai/generateEditorialPlan.ts:224](../utils/ai/generateEditorialPlan.ts:224)). Cap at 8.
- **Tip/take candidates:** `content_ideas` rows where `type = 'blog'` from the last 7 days, preferring `HOT` over `EVERGREEN`. Cap at 3.
- **Exclude alerts with unresolved fact-check flags.** If the underlying `alert` has `likely_wrong > 0` or `to_verify > 0` (per the chips shown on the 2026-04-21 brief), skip unless a human has resolved them. Newsletter posture is higher-trust than the daily brief.
- **Hard cap:** ~10 items total into the Sonnet prompt after both lists are merged.

## Schema — `newsletters` table

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `week_of` | date | Monday of the target week; unique |
| `subject` | text | Final Sonnet-generated subject |
| `body_html` | text | Rendered email HTML |
| `draft_json` | jsonb | Structured block source (hero, deals[], tip) |
| `status` | text | `scheduled \| held \| sent \| failed` |
| `scheduled_for` | timestamptz | Target send time (Fri 7am ET) |
| `sent_at` | timestamptz | Populated on send |
| `recipient_count` | int | Populated on send |
| `created_at` | timestamptz | default now() |

No `body_md`. Edits happen via structured JSON in admin UI, re-render HTML on save.

## Admin UI — `/admin/newsletter`

- Lists recent newsletters (scheduled / held / sent)
- Row click → preview + inline edit of `draft_json` blocks
- Buttons: "Hold this week" (scheduled → held), "Resume" (held → scheduled), "Send now" (manual override)
- Resend-test button: sends preview to Jill only

## Deferred to v2+

- **Canva MCP** magazine-style layouts (v1 is plain HTML)
- **Crazy-Wordle game** + `/play/[week]` landing page
- **Subject-line A/B**
- **`category` column on `content_ideas`** — only needed when game/tip split ships
- **Unsubscribe analytics / open tracking UI** — Resend captures it, no dashboard v1

## Blockers to resolve before "Send" works

- **Resend domain unverified** (per `memory/project_known_issues.md`). Blocks sending to real subscribers. Does not block draft generation or admin preview-to-Jill (which can go from any verified sender).

## Build order

1. Migration: create `newsletters` table
2. `/api/build-newsletter` route — writes a draft row (no Sonnet yet, just shape validation)
3. Sonnet prompt + generator logic (`buildNewsletter.ts`)
4. `buildNewsletterEmail.ts` renderer (HTML from `draft_json`)
5. Thu 6pm cron wiring in `vercel.json` + admin preview email via Resend
6. `/admin/newsletter` list + edit page
7. Fri 7am send cron + status transitions
8. End-to-end dry-run with Jill-only recipient list
9. Flip to real subscribers once Resend domain verified
